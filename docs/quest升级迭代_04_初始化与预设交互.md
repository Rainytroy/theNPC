# Quest 升级迭代 04：游戏初始化与预设交互体系

> **版本**: 1.0
> **状态**: 待实现
> **目标**: 将核心剧情的交互逻辑从“运行时动态生成”重构为“初始化预生成”，以提升游戏稳定性和剧情连贯性。

## 1. 核心理念 (Core Philosophy)

### 1.1 问题定义
完全依赖 LLM 实时生成的对话存在以下风险：
*   **不稳定性**: 关键剧情线索可能被忽略或幻觉。
*   **高延迟**: 每次简单交互都需要等待 API。
*   **逻辑割裂**: 任务道具与对话状态可能不同步。

### 1.2 解决方案
**“预设交互 (Pre-generated Interaction)”**：
在游戏进入 Runtime 之前，基于任务蓝图（Blueprint）一次性生成所有关键节点的对话、描述和交互逻辑。运行时引擎只负责“触发”和“展示”，而非“生成”。

---

## 2. 交互机制设计 (Interaction Design)

### 2.1 芯片交互系统 (Chips System)
当玩家满足特定条件（前置任务完成、时间/地点匹配）进入任务场景时，系统不再直接调用 LLM，而是直接弹出预设的交互选项（Chips）。

*   **标准选项**:
    *   **[调查] (Investigate)**: 触发环境探索流。
    *   **[什么也不做] (Do Nothing)**: 退出当前交互状态。

### 2.2 两种内容流 (Content Flows)

#### A. 调查流 (Investigation Flow)
*   **模式**: 纯描述性文本，无 NPC 参与。
*   **目的**: 通过环境描写提供沉浸式线索。
*   **示例**:
    > “大堂四下无人，你轻轻揭开酒坛的封泥。一张残片被藏在封泥中，上面标注了废弃道馆的位置。”
*   **设计原则**: 避免机械式指路，用游戏化的方式暗示下一步（如“也许正道人士会了解情况” -> 暗示去峨眉派）。

#### B. 对话流 (Dialogue Flow)
*   **模式**: 预设好的 NPC 与玩家对话剧本。
*   **道具交互**:
    *   在预设对话中嵌入**动作触发点 (Action Triggers)**。
    *   UI 显示 [展示道具] 或 [给予物品] 按钮。
    *   玩家点击后，系统执行 Item Owner 变更逻辑，并播放对应的反馈对话。

## 3. 资产构建管线 (Asset Generation Pipeline)

我们将“启动准备”阶段划分为三个严格串行的步骤。

### Step 1: 对白资产生成 (Dialogue Generation)
*   **输入**: 确认后的任务蓝图 (Quest Blueprint)。
*   **执行**: LLM 按照设计要求，为每个 Quest Node 生成对应的 `dialogue` 或 `investigation_desc`。
*   **存储策略**: **直接嵌入 `quests.json`**。
    *   理由: 保证“数据即逻辑”。引擎读取 Quest Node 时，内容即刻可用，无需二次索引。

### Step 2: 日程表生成 (Schedule Generation)
*   **依赖变更**: 必须在 **Step 1 完成后** 执行。
*   **逻辑输入**: NPC 人设 + **Quest 数据 (含对白)**。
*   **规则**:
    *   NPC 的行动必须服务于剧情。
    *   如果任务要求 NPC 在“午时”出现在“醉仙楼”，则日程表必须强制锁定该时段位置。

### Step 3: 开场白生成 (Intro Generation)
*   **输入**: 世界观 (World Bible) + 初始任务状态。
*   **输出**: 一段“星球大战”风格的滚动字幕。
*   **功能**:
    *   定调 (Mood Setting)。
    *   提供模糊的主线引导（如“去醉仙楼听听风声”），避免剧透但指明方向。

## 4. 启动体验 (Initialization UX)

用户确认蓝图后，进入**“世界构建中”**面板，通过可视化 UI 展示构建进度。

### 4.1 进度看板
*   **📜 史诗序章撰写中...** [========--]
*   **⚔️ 任务资源编织中...**
    *   主线剧情资产 [==========] ✅
    *   NPC 支线资产 [======----]
*   **📅 NPC 日程规划中...**
    *   小白的行程表 [===-------]
    *   老王的行程表 [----------]

### 4.2 进入游戏
*   仅当所有 CheckList 完成后，底部 **[进入世界 (Enter World)]** 按钮激活。
*   点击即开始 Game Loop，加载已生成好的完备数据。

## 5. 开发行动计划 (Action Plan)

### Phase 1: 后端服务构建 (Backend Services)
*   **Step 1.1: 资产填充服务 (QuestAssetEnricher)**
    *   **Prompt**: 编写 `prompts/asset_enricher.py`，负责生成“调查描述”和“对话脚本”。
    *   **Parser**: 实现清洗逻辑，将 LLM 输出解析并写入 `quests.json`，确保 JSON 格式安全。
*   **Step 1.2: 日程生成升级 (ScheduleGenerator)**
    *   修改日程生成逻辑，使其在规划前先读取 Quest 的强制时间/地点约束。
*   **Step 1.3: 开场白生成 (IntroGenerator)**
    *   实现简单的单次 LLM 调用接口。

### Phase 2: API 接口封装 (API Layer)
*   **Step 2.1: 状态管理**
    *   新增 `/genesis/initialize_world` 接口。
    *   设计初始化状态对象 (State Object) 以支持前端轮询进度。

### Phase 3: 前端交互实现 (Frontend UI)
*   **Step 3.1: 进度面板**
    *   改造 Genesis 页面，在蓝图确认后展示“初始化进度”。
*   **Step 3.2: 状态流转**
    *   实现 `Blueprint Confirmed` -> `Initializing` -> `Ready` 的状态机。
*   **Step 3.3: 进入游戏**
    *   绑定底部工具栏的 [进入世界] 按钮。
