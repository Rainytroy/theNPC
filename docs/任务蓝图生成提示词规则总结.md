# 任务蓝图生成提示词规则总结

## 📋 概述

Quest Writer 是一个 RPG 游戏任务生成 AI 引擎，包含**主任务（Main Quest）**和**支线任务（Side Quest）**两种生成模式。本文档详细分析两种任务类型的生成规则、约束条件及其异同点。

---

## 🎯 一、核心任务范式（2种节点类型）

两种任务类型都使用相同的两种核心节点：

### 1. **dialogue (对话节点)**
- **核心功能**：与 NPC 进行人际互动和物品交换
- **必需参数**：`target_npc_id`
- **可用条件**：
  - `affinity`（必需）：好感度要求
  - `location`（可选）：地点限制
  - `time`（可选）：时间限制
  - `item`（可选）：展示或给予物品
- **奖励**：可包含物品奖励（从 NPC 处获得物品）
- **物品归属规则**：如果此节点奖励新物品，该物品的 `owner_id` **必须**设置为该节点的 `target_npc_id`

### 2. **investigate (调查节点)**
- **核心功能**：与环境/场景互动，环境叙事
- **必需参数**：`location_id`
- **约束**：`target_npc_id` 应为 null
- **可用条件**：
  - `time`（可选）：时间限制
  - `item`（可选）：使用物品开启/修复某物
- **奖励**：可选物品奖励（发现某物）
- **物品归属规则**：如果此节点奖励新物品，该物品的 `owner_id` **必须**设置为该节点的 `location_id`

## 🔐 二、关键约束规则（两种任务共有）

### 2.1 数量限制规则

| 限制项 | 主任务（Main Quest） | 支线任务（Side Quest） |
|--------|---------------------|----------------------|
| **任务数量** | 必须生成 **1个** | 必须生成 **1个** |
| **节点数量** | 最多 **7个节点** | 最多 **5个节点** |
| **新物品数量** | 最多创建 **2个新物品** | 最多创建 **2个新物品** |

### 2.2 物品归属约束（ITEM OWNERSHIP）

这是一个**强制性规则**，用于确保物品的清晰归属：

#### 🚨 **新物品（new_items）归属规则 - 强制性**

每个新创建的物品**必须**指定 `owner_id`，**不能为 null**：

| 节点类型 | owner_id 设置规则 |
|---------|------------------|
| **dialogue 节点奖励** | owner_id = target_npc_id（给予该物品的 NPC）|
| **investigate 节点** | owner_id = location_id（发现该物品的地点）|

**预生成自检流程**（输出 JSON 前必须执行）：
1. 列出所有在 `new_items` 中创建的新物品
2. 对每个新物品：验证 `owner_id` 已设置且**不为 null**
3. 对现有物品（来自 Available Items）：可选择添加到 `item_ownership_updates` 以指定所有者
4. 如果任何新物品缺少 `owner_id`，**必须修复**：
   - 为持有它的 NPC 分配（dialogue）
   - 为找到它的地点分配（investigate）
   - **绝不**输出 null/空的 owner_id

#### ✅ **现有物品归属声明 - 推荐性**

- 如果使用 **Available Items** 中的现有物品，**推荐**（但非强制）通过 `item_ownership_updates` 数组指定其所有者
- 这有助于系统知道哪个 NPC 或地点在任务中持有该物品

### 2.3 逻辑一致性规则

#### a) 🔑 **获取前置原则（Receive Before Use）- 强制检查**

**核心规则**：如果任何节点在条件中要求玩家**展示（show）**或**给予（give）**物品，则**同一任务**中的**前置节点**必须通过奖励授予该物品（`"action": "receive"`）。

**自检流程**（输出 JSON 前）：
1. 列出所有在条件中使用 `action="show"` 或 `action="give"` 的 item_id
2. 对每个 item_id，向前搜索该任务的前置节点
3. 验证至少有一个前置节点在奖励中包含该 item_id 且 `action="receive"`
4. 如果未找到，**必须修复**：
   - 在任务前面添加一个 receive 节点，或
   - 移除该物品条件，使用不同方法

**示例：**

❌ **错误**：
```json
Node 3: condition { "item_id": "wine", "action": "give" }
// 但没有前置节点奖励 wine
```

✅ **正确**：
```json
Node 1: rewards [{ "type": "item", "params": { "item_id": "wine", "action": "receive" }}]
Node 3: condition { "type": "item", "params": { "item_id": "wine", "action": "give" }}
```

#### b) 🎯 **单一物品流转（Single Item Flow）**

**主任务规则**：
- 每个物品只能在**一个**任务流中使用
- **不要**在多个任务（主线+支线）中使用同一物品
- **不要**在不同任务中多次请求同一物品

❌ **错误**：主任务使用 item_001，支线任务也使用 item_001
✅ **正确**：主任务使用 item_001，支线任务使用 item_002 或 item_003

**支线任务规则**：
- **不要**使用 "Used Items from Main Quest" 列表中的物品
- 每个物品应只出现在一个任务中（主线**或**支线）

### 2.4 条件系统规则

#### 🤝 **Affinity（好感度）系统**
```json
{ "type": "affinity", "params": { "value": N } }
```

**N 值范围：0-4**
- **Lv 0**: Stranger（陌生人）
- **Lv 1**: Acquaintance（认识）
- **Lv 2**: Friend（朋友）
- **Lv 3**: Trust（信任）
- **Lv 4**: Love（深爱）

任务节点使用此条件根据关系等级进行门槛控制。

#### ⏰ **Time（时间）系统**
```json
{ "type": "time", "params": { "start": "HH:MM", "end": "HH:MM" } }
```

#### 📦 **Item（物品）系统**

**物品条件**（在 conditions 中）：
```json
{ "type": "item", "params": { "item_id": "xxx", "action": "show|give" } }
```
- `show`：玩家展示物品（不消耗）
- `give`：玩家给予物品（消耗）

**物品奖励**（在 rewards 中）：
```json
{ "type": "item", "params": { "item_id": "xxx", "action": "receive" } }
```
- `receive`：玩家获得物品

### 2.5 资产使用与创建规则

#### 优先级：
1. **严格优先**使用 **Available Items**（可用物品列表）
2. 如果故事绝对需要列表中不存在的物品，可以**创建**（受限于最多 2 个新物品）

#### 创建新物品要求：
- 添加到 `new_items` 列表
- ID 必须以 `item_new_` 开头
- **必须**遵守上述物品归属规则（owner_id 不能为 null）
- 在任务节点中使用该新 ID

## 🔄 三、主任务 vs 支线任务 - 异同对比

### 3.1 相同点 ✅

| 方面 | 描述 |
|-----|------|
| **节点类型** | 都使用 dialogue、investigate 两种节点 |
| **物品归属规则** | 都必须遵守新物品 owner_id 强制规则 |
| **逻辑一致性** | 都必须遵守"获取前置原则"和"单一物品流转" |
| **条件系统** | 都使用相同的 affinity、time、item 条件系统 |
| **新物品限制** | 都限制最多创建 2 个新物品 |
| **JSON 输出** | 都要求严格的 JSON 格式输出 |
| **自检流程** | 都要求输出前执行物品归属和逻辑一致性检查 |

### 3.2 不同点 ❌

| 对比维度 | 主任务（Main Quest） | 支线任务（Side Quest） |
|---------|---------------------|----------------------|
| **任务定位** | 编织世界观和可用物品成为引人入胜的主线叙事弧 | 为特定 NPC 创建个人叙事弧，加深关系 |
| **输入数据** | World Bible, NPC Roster, Available Items, Locations, Time Config | Target NPC (helper/blocker), World Bible, Available Items, Locations, **Used Items from Main Quest** |
| **节点数量上限** | **最多 7 个节点** | **最多 5 个节点** |
| **任务焦点** | 整体世界的主要目标 | 单个 NPC 的个人故事 |
| **NPC 角色** | 涉及多个关键 NPC | 聚焦一个目标 NPC（helper 或 blocker） |
| **物品冲突规避** | 需确保物品不在多个任务间重复 | **必须避免**使用主任务已用的物品（接收明确的排除列表）|
| **任务 ID 前缀** | `mq_01` | `sq_npcname_01` |
| **任务类型** | `"type": "main"` | `"type": "side"` |
| **目标 NPC 字段** | 无 `target_npc_id` 字段（任务级别）| 有 `target_npc_id` 字段指向目标 NPC |
| **设计逻辑重点** | 宏大叙事、世界拯救、多线索整合 | 个人请求、秘密分享、情感深化 |
| **好感度起点** | 根据剧情需要灵活设置 | Helper 任务通常从 Lv 1 开始，Blocker 从 Lv 0 开始 |

## 🎓 五、核心设计理念总结

### 5.1 物品流转的严格性
- **物品归属清晰**：每个物品都有明确的持有者（NPC 或地点）
- **获取逻辑合理**：玩家使用物品前必须先获得
- **避免资源冲突**：主线和支线不共享物品，确保任务独立性

### 5.2 任务渐进性设计
- **好感度门槛**：通过 affinity 系统控制任务解锁
- **节点依赖链**：前置节点完成后才能解锁后续节点
- **时间和地点限制**：增加游戏真实感和策略性

### 5.3 创作自由度 vs 约束平衡
- **优先使用现有资产**：保持世界一致性
- **允许创建新物品**：但限制数量（最多2个），避免资产膨胀
- **强制逻辑检查**：确保生成的任务可执行且无逻辑漏洞

### 5.4 主线与支线的协同设计
- **主线**：宏大叙事，多 NPC 互动，较长的任务链（最多7节点）
- **支线**：聚焦个人，深化关系，较短的任务链（最多5节点）
- **资源隔离**：支线明确避开主线已用物品，确保不冲突

### 5.5 简化的节点系统
- **只有2种节点类型**：dialogue（NPC互动+物品交换）和 investigate（环境探索）
- **dialogue 节点整合功能**：既可以对话，也可以交换物品，简化了原本的 collect 节点
- **示例即规范**：AI 通过示例学习输出格式，不生成示例中未出现的字段（如对话文本）

## ✅ 六、质量保证检查清单

在生成任务 JSON 前，AI 必须完成以下自检：

### 📋 物品归属检查
- [ ] 所有 `new_items` 都有 `owner_id` 且不为 null
- [ ] dialogue 节点的新物品：owner = target_npc_id
- [ ] investigate 节点的新物品：owner = location_id

### 📋 逻辑一致性检查
- [ ] 所有 show/give 的物品都在前置节点中 receive
- [ ] 支线任务未使用主任务的物品列表中的物品
- [ ] 每个物品只在一个任务中使用

### 📋 数量限制检查
- [ ] 主任务节点数 ≤ 7，支线节点数 ≤ 5
- [ ] 新物品数量 ≤ 2
- [ ] 只生成 1 个任务

### 📋 格式规范检查
- [ ] 所有必需字段都已填写
- [ ] JSON 格式正确，无语法错误
- [ ] ID 命名符合规范（mq_ / sq_ / item_new_）
- [ ] affinity 值在 0-4 范围内
- [ ] time 格式为 HH:MM
- [ ] 节点类型只有 dialogue 或 investigate

## 🎯 七、结论

Quest Writer 的提示词设计展现了**结构化创意**的理念：

1. **严格的规则框架**：通过数量限制、归属约束、逻辑检查确保生成质量
2. **灵活的创作空间**：在规则内允许创建新物品和多样化的任务设计
3. **主支线分工明确**：主线宏大叙事，支线深化关系，相互独立又协同
4. **强制预检机制**：AI 在输出前必须自我验证，减少逻辑错误
5. **简化的节点系统**：从3种节点类型简化为2种，dialogue 节点整合了与 NPC 的所有互动（对话+物品交换）
6. **示例驱动设计**：通过精心设计的 JSON 示例控制 AI 输出，AI 不会生成未见过的字段

这套系统平衡了**游戏设计的一致性**与**AI 创作的灵活性**，为开放世界 RPG 提供了可靠且可控的任务生成方案。
